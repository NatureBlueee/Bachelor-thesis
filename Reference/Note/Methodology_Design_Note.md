# 第一部分：核心概念理解

## 1.1 什么是"演绎法"（Deductive Approach）？

### 通俗解释

想象你是一个侦探：
- **演绎法**：你有一个理论（"凶手总是会回到犯罪现场"），然后你去观察看这个理论是否成立
- **归纳法**：你观察了很多案例，然后总结出一个规律

### 在研究方法中的含义

| 方法 | 起点 | 过程 | 目的 |
|------|------|------|------|
| **演绎法** (Deductive) | 从理论/框架开始 | 用理论去"看"数据 | 验证/扩展理论 |
| **归纳法** (Inductive) | 从数据开始 | 从数据中"生成"理论 | 建构新理论 |

### 我们的研究是什么？

**我们是"演绎为主，归纳为辅"**：
- **演绎的部分**：我们有RDT理论框架和命题P1-P4，用它们来指导我们"看"什么
- **归纳的部分**：我们保持开放，如果数据告诉我们理论没覆盖到的东西，我们也记录

**用原文的话说**（Roberts et al., 2021）：
> "An initial coding framework was developed with reference to the P-TAF factors... During this process, the coders **remained open for additional or alternative psychological codes**."

翻译：我们有预设的编码框架，但分析时对新的发现保持开放。

## 1.2 什么是"主题分析"（Thematic Analysis）？

### 通俗解释

主题分析就是：**从一堆访谈文字中找到反复出现的"主题"（模式）**。

想象你收到100封顾客投诉信，你会：
1. 读所有信
2. 发现很多人抱怨"发货慢"
3. 发现很多人抱怨"客服态度差"
4. 把这些归类成"主题"

主题分析就是这个过程的学术版本。

### Braun & Clarke (2006) 六阶段流程

| 阶段 | 你做什么 | 产出 |
|------|----------|------|
| **1. 熟悉数据** | 反复阅读访谈记录 | 初步想法 |
| **2. 生成初始编码** | 标记有意思的片段 | 编码列表 |
| **3. 寻找主题** | 把编码归类成主题 | 候选主题 |
| **4. 审视主题** | 检查主题是否合理 | 修订后的主题 |
| **5. 定义主题** | 给每个主题命名和定义 | 主题描述 |
| **6. 撰写报告** | 写出分析结果 | 论文章节 |

### 演绎主题分析 vs 归纳主题分析

| 类型 | 编码来源 | 适用情况 | 我们的选择 |
|------|----------|----------|------------|
| **演绎主题分析** | 从理论框架预设 | 已有理论想验证 | ✅ 我们用这个 |
| **归纳主题分析** | 从数据中涌现 | 探索新现象 | 部分使用 |

## 1.3 什么是"编码本"（Codebook）？

### 通俗解释

编码本就是一个**"翻译词典"**：
- 告诉你什么样的话应该被标记为什么类别
- 确保不同的人看同一段话会给相同的编码

### 编码本的结构

一个好的编码本包含：

| 组成部分 | 说明 | 示例 |
|----------|------|------|
| **编码名称** | 这个类别叫什么 | "AI情感 - 自信" |
| **定义** | 这个编码的具体含义 | "受访者表达对使用AI的信心和积极情感" |
| **纳入标准** | 什么情况应该用这个编码 | 当受访者说"我觉得自己用得挺好的" |
| **排除标准** | 什么情况不应该用 | 单纯描述技能，没有情感表达 |
| **示例** | 典型的语句 | "我比同事更敢尝试新的AI工具" |

## 1.4 什么是"数据结构"（Data Structure）？

### 通俗解释

数据结构是一张**"从原始数据到理论概念的地图"**，展示：
- 受访者的原话 → 初步编码 → 归纳主题 → 理论概念

### 为什么重要？

Gioia et al. (2013) 说：
> "No data structure; know nothing."
> 没有数据结构，就不知道你的发现从哪来的。

它展示了你的分析过程是**透明的、可追溯的**。

### 数据结构的样子

```
受访者原话（第一阶编码）     →    主题（第二阶编码）    →    理论维度
─────────────────────────     ─────────────────────     ─────────
"我知道ChatGPT能做什么"        AI工具功能认知           AI知识
"我了解AI的局限性"              AI能力边界认知           AI知识
"我比我领导更会用prompt"        相对技能感知             感知资源优势
"我觉得领导需要我帮他学"        被需要感                 感知资源优势
"我主动跟领导说可以用AI"        主动建议行为             向上影响行为
```

## 1.5 什么是"理论饱和"（Theoretical Saturation）？

### 通俗解释

当你访谈到第10个人，发现他说的和前9个人说的差不多、没有新东西了——这就是饱和了。

### 实操标准

**Doargajudhur & Hosanoo (2022) 的做法**：
> "Saturation is the point **where no new discoveries or evolution of the categories are yielded** through the coding of data."

翻译：当新的访谈不再产生新的发现或新的编码类别时，就饱和了。

### 我们的标准

- 目标样本：15-20人
- 饱和检查：每5个访谈后评估是否还有新主题出现
- 停止条件：连续3个访谈没有产生新的编码/主题

# 第二部分：我们的研究设计

## 2.1 研究问题

**核心问题**：AI素养如何使Z世代员工影响他们的上级？

**子问题**：
1. Z世代员工如何感知自己的AI素养？（AI知识、AI情感、AI思维）
2. 这种素养如何转化为"感知资源优势"？
3. 感知资源优势如何促发向上影响行为？
4. 上级开放性如何影响这个过程？

---

## 2.2 理论框架与命题

### 核心理论

**资源依赖理论（RDT）**的核心逻辑：
> 谁掌握他人需要的资源，谁就有影响力。

**应用到我们的研究**：
> Z世代员工的AI素养是一种"资源"。如果上级需要这种资源但自己不具备，员工就获得了影响力基础。

### 命题（Propositions）

> **注意**：在质性研究中，我们用"命题"不用"假设"。命题是**分析指南**，不是**统计检验对象**。

| 命题 | 内容 | 对应编码类别 |
|------|------|--------------|
| **P1** | AI素养（知识+情感+思维）塑造员工对资源优势的感知 | AI Literacy → Perceived Resource Advantage |
| **P2** | 感知资源优势促发向上影响行为 | Perceived Resource Advantage → Upward Influence |
| **P3** | 上级开放性影响向上影响行为的发生和结果 | Supervisor Openness (Moderating) |

### 与范例论文的对比

| 我们的命题 | 类似范例 | 写法特点 |
|------------|----------|----------|
| P1: AI素养→感知资源优势 | Fréour (2021) P1: 技术→自主性感知 | 用"shapes"不用"positively affects" |
| P2: 感知资源优势→向上影响 | Roberts (2021) 框架→行为 | 中性表述，让数据说话 |

**初步猜想**：

### 步骤3：开发编码本（预设 + 开放）

**预设编码**（来自理论）：

| 一级类别 | 二级类别 | 定义 | 可能的原话 |
|----------|----------|------|------------|
| AI知识 | 功能认知 | 了解AI能做什么 | "我知道它能帮我写代码" |
| AI知识 | 边界认知 | 了解AI的局限 | "但它经常会胡说" |
| AI情感 | 使用自信 | 对自己使用AI的信心 | "我用得挺溜的" |
| AI情感 | 学习热情 | 对探索AI的兴趣 | "我喜欢研究新功能" |
| AI思维 | 批判评估 | 质疑AI输出 | "我会检查它的答案" |
| 感知资源优势 | 相对技能 | 觉得比他人强 | "我比我领导懂" |
| 感知资源优势 | 被需要感 | 感到被依赖 | "他们有问题会问我" |
| 向上影响 | 主动建议 | 自发提出想法 | "我主动跟领导说..." |
| 向上影响 | 知识分享 | 教授AI使用 | "我教他怎么用" |
| 上级开放性 | 接纳态度 | 愿意听建议 | "他很愿意学" |
| 上级开放性 | 实际采纳 | 真的采用建议 | "后来他真的用了" |

- [ ] 可能出现：组织文化因素

## 2.4 访谈设计与编码框架的对应

### 访谈提纲逻辑

| 访谈模块 | 对应编码类别 | 问题设计思路 |
|----------|--------------|--------------|
| 暖场 + 背景 | 基本信息 | 了解工作情境 |
| AI使用经历 | AI素养（三维度） | 先描述，再感受 |
| 与他人比较 | 感知资源优势 | 自然引出相对感知 |
| 影响经历 | 向上影响行为 | 请讲具体故事 |
| 上级反应 | 上级开放性 | 描述互动过程 |
| 开放补充 | 开放编码 | 捕捉未预料主题 |

### 访谈问题设计原则

1. **从描述开始**：先问"是什么"，再问"怎么感觉"
2. **用故事探索**：请讲一个具体的经历
3. **避免引导**：不问"你是不是比领导强？"而问"你和领导的AI使用有什么不同？"
4. **追问细节**：为什么？那时候你怎么想的？后来呢？

## 2.5 数据分析流程

```
访谈录音
    ↓
转录成文字
    ↓
熟悉数据（反复阅读）
    ↓
初始编码（用预设编码本 + 开放编码）
    ↓
整理编码（检查一致性）
    ↓
寻找主题（编码→主题）
    ↓
审视主题（与理论对话）
    ↓
构建数据结构（第一阶→第二阶→理论维度）
    ↓
撰写发现章节
```

## 2.6 严谨性保障

| 质量标准 | 策略 | 具体做法 |
|----------|------|----------|
| **可信度** (Credibility) | 成员检查 | 让部分受访者看初步发现 |
| **可转移性** (Transferability) | 厚描述 | 详细描述情境和过程 |
| **可靠性** (Dependability) | 审计轨迹 | 记录所有分析决策 |
| **可确认性** (Confirmability) | 引用原话 | 用直接引语支撑解读 |
